\begin{abstract}
SoundCode è un'applicazione sviluppata in Scala che permette di creare pattern musicali tramite una mini-DSL testuale, ispirata al paradigma del live coding musicale di Strudel/TidalCycles. L'utente scrive espressioni musicali all'interno di una GUI; il sistema le analizza, le interpreta, genera eventi musicali temporizzati e li riproduce tramite sintetizzatore MIDI e, in forma essenziale, tramite sampler.

L'obiettivo principale del progetto non è replicare integralmente Strudel, ma realizzare una versione ridotta e coerente con il contesto del corso PPS, valorizzando la modellazione funzionale del dominio, la composizione di pattern, il parsing di una DSL e la separazione tra interfaccia utente, modello del dominio, scheduling ed esecuzione sonora.
\end{abstract}

\section{Introduzione}
% ============================================================

SoundCode nasce come ambiente minimale di live coding musicale. L'applicazione consente all'utente di descrivere pattern ritmici e melodici attraverso un linguaggio testuale semplice, progettato specificamente per il progetto. Tali espressioni vengono trasformate in una rappresentazione interna, successivamente convertita in eventi musicali temporizzati e riproducibili.

Il sistema è ispirato a strumenti come Strudel e TidalCycles, ma viene progettato con un perimetro funzionale ridotto. L'attenzione principale è posta su:
\begin{itemize}
\item progettazione di una mini-DSL musicale;
\item parsing e interpretazione del testo scritto dall'utente;
\item modellazione funzionale di note, pause, pattern, eventi e trasformazioni;
\item generazione di una timeline di eventi musicali;
\item riproduzione tramite sintetizzatore MIDI e supporto essenziale a sampler;
\item interazione tramite GUI.
\end{itemize}

Un esempio preliminare di codice SoundCode potrebbe essere il seguente:

\begin{lstlisting}[style=soundcode]
bpm 120
instrument piano
play [c4 d4 e4 g4]
\end{lstlisting}

Oppure, con una trasformazione:

\begin{lstlisting}[style=soundcode]
play repeat 2 [c4 d4 e4 g4]
\end{lstlisting}

Questi esempi sono da considerarsi provvisori: la sintassi definitiva della DSL sarà definita e raffinata durante le prime iterazioni di sviluppo.

\subsection{Modello di dominio}

I concetti principali del dominio sono:

\begin{description}
\item[Pattern] struttura musicale astratta che descrive una sequenza, una sovrapposizione o una trasformazione di elementi musicali.
\item[Nota] evento musicale caratterizzato da altezza, ottava, durata, volume e strumento.
\item[Pausa] intervallo temporale privo di suono.
\item[Evento musicale] elemento temporizzato generato a partire da un pattern, caratterizzato da istante di inizio, durata e contenuto sonoro.
\item[Timeline] sequenza ordinata di eventi musicali pronti per essere schedulati.
\item[Trasformazione] operazione applicabile a un pattern, ad esempio ripetizione, concatenazione, sovrapposizione o variazione della velocità.
\item[Scheduler] componente che riceve una timeline e si occupa di eseguire gli eventi musicali nel tempo corretto.
\item[Sintetizzatore MIDI] componente responsabile della riproduzione delle note tramite MIDI.
\item[Sampler] componente responsabile della riproduzione di campioni sonori predefiniti.
\item[Sessione] stato corrente dell'applicazione, comprendente codice scritto dall'utente, parametri musicali e pattern in esecuzione.
\end{description}

\section{Design architetturale}
% ============================================================

\subsection{Visione generale}

L'architettura prevista per SoundCode è organizzata in componenti con responsabilità distinte. Una prima ipotesi è una struttura ispirata a MVC/layered architecture, in cui la GUI non contiene logica musicale, ma interagisce con un controller o application service che coordina parser, modello, scheduler e audio engine.

\subsection{Componenti principali}

\begin{longtable}{p{0.25\textwidth} p{0.65\textwidth}}
\toprule
\textbf{Componente} & \textbf{Responsabilità} \
\midrule
GUI & Permette all'utente di scrivere codice SoundCode, avviare/fermare l'esecuzione e visualizzare messaggi di errore o stato. \
Application Controller & Coordina le richieste provenienti dalla GUI e invoca parser, pattern engine, scheduler e audio engine. \
Parser & Analizza la DSL testuale e produce una rappresentazione interna del programma musicale. \
Domain Model & Definisce le entità principali del dominio: note, pause, pattern, eventi, timeline e trasformazioni. \
Pattern Engine & Interpreta i pattern e applica trasformazioni come repeat, slow, fast, concatenazione e sovrapposizione. \
Scheduler & Converte la timeline in eventi temporizzati da eseguire nel momento corretto. \
MIDI Engine & Riproduce le note tramite sintetizzatore MIDI. \
Sampler Engine & Riproduce campioni sonori predefiniti. \
Persistence & Gestisce eventuale salvataggio e caricamento dei pattern. \
Testing & Contiene test unitari e di integrazione per validare parser, modello, trasformazioni e scheduling. \
\bottomrule
\end{longtable}

\subsection{Flusso principale di esecuzione}

Il flusso principale previsto è il seguente:
\begin{enumerate}
\item l'utente scrive un pattern nella GUI;
\item l'utente preme Play;
\item il controller recupera il testo inserito;
\item il parser analizza il testo;
\item se il parsing fallisce, la GUI mostra l'errore;
\item se il parsing ha successo, viene prodotto un pattern interno;
\item il pattern engine applica eventuali trasformazioni;
\item lo scheduler genera una timeline di eventi;
\item il MIDI engine e/o il sampler engine riproducono gli eventi.
\end{enumerate}

\section{Design di dettaglio}
% ============================================================

\subsection{Mini-DSL}

La sintassi definitiva sarà definita durante i primi sprint. Una prima idea prevede:

\begin{lstlisting}[style=soundcode]
play [c4 d4 e4 g4]
play repeat 2 [c4 d4 e4]
play slow 2 [c4 d4]
play fast 2 [c4 d4]
\end{lstlisting}

Possibili elementi della DSL:

\begin{itemize}
\item note: \texttt{c4}, \texttt{d4}, \texttt{e4};
\item pause: \texttt{-};
\item sequenze: \texttt{[c4 d4 e4]};
\item trasformazioni: \texttt{repeat}, \texttt{slow}, \texttt{fast};
\item parametri: \texttt{bpm}, \texttt{instrument}, \texttt{volume}.
\end{itemize}

\subsection{Modello dati preliminare}

Una possibile modellazione in Scala potrà basarsi su ADT simili ai seguenti:

\begin{lstlisting}[style=soundcode]
sealed trait MusicalElement
case class Note(name: Pitch, octave: Int, duration: Duration) extends MusicalElement
case class Rest(duration: Duration) extends MusicalElement

sealed trait Pattern
case class Single(element: MusicalElement) extends Pattern
case class Sequence(patterns: List[Pattern]) extends Pattern
case class Stack(patterns: List[Pattern]) extends Pattern
case class Repeat(times: Int, pattern: Pattern) extends Pattern
case class Slow(factor: Double, pattern: Pattern) extends Pattern
case class Fast(factor: Double, pattern: Pattern) extends Pattern

case class MusicalEvent(
start: Time,
duration: Duration,
element: MusicalElement
)
\end{lstlisting}

Questo codice è da considerarsi puramente indicativo e verrà aggiornato in base alle scelte effettive di implementazione.
